# 二、项目背景与目标

## 1. 项目背景

检e通"居配试验检测管理系统"是公司面向居配工程试验检测业务的核心作业工具，覆盖"建工单→审批→派发→现场核查→设备检测→包裹生成与导出→结算管理"的业务全链路。

2.4系统经多轮试用暴露出以下核心问题：

- **流程过度线性化**：系统强制所有环节串行完成，与现场并行作业的实际需要严重冲突。例如：所有站房的建设情况必须全部确认后才能启动任何一个站房的安措流程，导致全队空等。
- **现场作业效率低**：双重录入（拍照+手填）、同类设备重复前置操作、仪器入场逐一拍照阻塞主流程等问题，使关键操作路径冗长。
- **无网环境数据丢失**：地下室等无信号环境下，APP无法正常拍照和提交数据；更严重的是，无网时拍摄的照片在恢复网络后上传失败，已在真实省检现场造成数据丢失。
- **报告返工率高**：报告黑盒生成、检测标准参数不匹配（如SCB12/14标准混用）等问题导致报告产出后仍需大量人工修正。
- **系统扩展性差**：检测逻辑硬编码、业务规则分散在数据库存储过程/Java/JS代码中，新增设备类型或检测项的开发成本极高。

基于以上问题，公司决定在已有技术底座与自研低代码平台基础上，对系统进行全面重构。

## 2. 产品目标

- 2026年6月30日前，完成检e通移动端App与Web后台的全流程重构。系统按**现场模块、任务模块、结算模块**三大核心模块划分，一期优先将现场模块做深做透，彻底解决一线试验人员的现场作业痛点；二期补齐任务管理与结算管理，实现业务全闭环。
- 消除2.4系统核心痛点：以现场作业场景为突破口，重点解决双重录入、批量作业不畅、报告返工等阻碍一线效率的问题，优化关键操作路径。
- 构建可复用的居配检测业务平台：通过配置化的检测流程与报告模板架构，使系统不局限于省检单一场景，具备快速适配市检、预检等多种检测类型的能力。

## 3. 产品成功指标

| 指标    | 目标值                       | 度量方式       |
| :---- | :------------------------ | :--------- |
| 交互效率  | 关键操作路径点击次数较2.4系统减少**40%以上** | 逐流程量化对比    |
| 系统稳定性 | 核心业务流程**零阻断**             | 测试+试用期监控   |
| 用户满意度 | ≥**80%**的一线试验人员试用满意       | SUS问卷      |
| 报告质量  | 生成报告可直接交付，无需大量人工二次修改      | 试用期报告返工率统计 |
| 无网保护  | 无网环境下**不丢数据、不阻断操作**     | 有网/无网场景专项测试   |

# 四、需求范围与边界

## 1. 范围内（Scope In）

### V3.0（第一期） — 现场模块深耕

> 目标：彻底解决一线试验人员的现场作业痛点，将现场模块做深做透；同时验证产品设计并采集使用数据。

- **轻量工单创建与派发**（基于低代码表单+现有流程引擎，支持产品经理直接创建工单并派发给检测队工作负责人，不含审批流转。作为现场模块的前置输入，确保V3.0可独立闭环运行）
- **现场核查全流程**（作业前准备、现场准备、建设情况确认、核查分工、核查执行、核查确认、抽检对象确认、安措执行、开工会、人员分工）
- **检测数据采集**（覆盖全部设备类型的检测录入、批量操作、拍填分离、增补检测对象、检测跳过等）
- **报告包裹生成与导出**（数据自动汇总、报告确认前置、关联设备报告合并输出）
- **数据配置中台**（设备类型-检测项-阈值动态配置、设备铭牌数据字典、检测标准参数库、报告模板管理、业务规则集中化管理）
- **系统基础底座**（用户管理、权限管理、组织架构）
- **无网数据保护**（本地优先写入 + 恢复网络自动上传 + 表单草稿断点续填）

### V3.1（第二期） — 任务模块 + 结算模块

> 目标：补齐计划管理与结算管理，实现完整业务闭环，扩大试用范围。

- **任务模块**（在V3.0轻量工单基础上叠加审批流转、退回修改、计划管理全模块、消息通知）
- **结算管理**（以康乐内部部门经理为核心视角，满足80%结算需求）
- **支撑配套**（项目总览/仪表盘、资产管理、辅助管理、审计与日志）
- V3.0优化增强项

> 约束：结算模块优先级低于任务模块，当开发资源冲突时优先保障任务链路。

# 五、业务状态机

> 描述一个项目从工单创建到最终完成的全生命周期状态流转。V3.0的状态机设计遵循两个原则：①主链路严格串行保证数据完整性；②非关键确认环节（核查确认）不阻塞主链路，允许旁支并行。

## 1. 工单级状态机（V3.0）

```
                        ┌─────────────────────────────────────┐
                        │          V3.1 新增                   │
                        │  草稿(draft) → 待审批(pending_approve)│
                        │       ↓              ↓               │
                        │     提交           通过 / 拒绝        │
                        └───────┬─────────────┬───────────────┘
                                │             │
                                ▼             ▼
项目经理创建并提交 ──→ 已派发(dispatched) ──→ 已接收(accepted)
                        项目经理提交即派发     工作负责人确认接收
```

## 3. 状态机设计要点

| 要点 | 说明 |
|:---|:---|
| **主链路严格串行** | 作业前准备→现场准备→建设情况确认→核查分工→核查执行→抽检对象确认，每一步必须前一步完成后才能进入 |
| **核查确认非阻塞** | 核查已完成后，主链路直接推进到抽检对象确认。工作负责人的核查确认是旁支操作，不阻塞安措和检测的执行 |
| **安措是检测的强制前置** | 按站房粒度管控：某站房安措完成→该站房检测解锁。未安措的站房，检测入口锁死，任何人无法绕过 |
| **按站房粒度并行** | 安措和检测阶段，各站房独立推进，互不阻塞。1号站房检测中的同时，3号站房可能还在安措 |
| **所有已编写环节提交后不可修改** | V3.0设计原则：每个环节提交即锁定，不做回退。数据一致性优先于操作灵活性 |
| **过门页不保留中间状态** | 作业前准备、现场准备、建设情况确认、核查分工：退出重进需重新操作（操作量小，可接受） |
| **核查执行必须保留中间状态** | 照片拍一张存一张（现场实拍不可重来），与过门页原则不同 |

#### 1.1.0 轻量工单创建与派发

| 维度 | 说明 |
|:---|:---|
| **角色** | 项目经理（创建派发）、工作负责人（确认接收） |
| **时机** | 检测日前一天，项目经理创建工单并派发给检测队 |
| **做什么** | 填写精简表单完成工单创建与派发，为后续全部现场环节建立项目级数据基础 |
| **版本归属** | V3.0（现场模块前置依赖） |

**业务定位**：轻量工单是整个现场模块的入口。项目经理在出发前创建工单并直接指派给检测队工作负责人，产出项目基础数据和检测队伍/人员指派。工作负责人确认接收后，进入现场核查流程。

**角色与管理颗粒度**：

| 角色 | 管理颗粒度 | 职责 |
|:---|:---|:---|
| 项目经理 | 队伍级 | 选哪支队伍去、队伍间借人调配、确认工作负责人。不关心队内怎么分工 |
| 工作负责人 | 任务级 | 到场后把核查/检测任务分给具体的人，现场灵活调配 |

**表单字段定义（单页表单，基础信息6字段+项目规模4字段+派发区3字段）**：

*基础信息区（6字段）*：

| # | 字段 | 组件类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---|:---|:---|:---|
| 1 | 项目名称 | 文本输入 | ✓ | — | 自由文本 |
| 2 | 项目类型 | 下拉单选 | ✓ | 市检 | 省检/预检/市检 |
| 3 | 传票号 | 文本输入 | ✓ | — | 英文+数字格式（正则：`^[A-Za-z0-9]+$`），全局唯一校验 |
| 4 | 行政区划 | 文本输入 | ✓ | — | 手动输入，如"江苏省盐城市亭湖区" |
| 5 | 检测日期 | 日期选择 | ✓ | — | 不设默认值，只能选单天，不限日期范围 |
| 6 | 低压新增项目 | 下拉单选 | ✓ | 否 | 是/否，选"是"时下游电源核查走简化模式 |

*项目规模区（4个独立字段，视觉分组）*：

| # | 字段 | 组件类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---|:---|:---|:---|
| 7a | 站数量 | 数字输入 | ✓ | 0 | 开关站 |
| 7b | 房数量 | 数字输入 | ✓ | 0 | 配电室 |
| 7c | 合建站房数量 | 数字输入 | ✓ | 0 | 合建开关站 |
| 7d | 箱变数量 | 数字输入 | ✓ | 0 | 箱式变电站 |

> 校验规则：站 + 房 + 合建 + 箱变 > 0（至少填一项），否则拦截提示"请至少填写一项项目规模"。提交后后端通过 `genStationRoomInfoNew()` 按数量自动生成站房记录，作为下游"建设情况确认"的初始数据。

*派发区（3字段，含自动带出）*：

| # | 字段 | 组件类型 | 必填 | 交互逻辑 |
|:---|:---|:---|:---|:---|
| 8 | 检测队伍 | 成员单选 | ✓ | 从组织架构中选择，选择后自动带出默认工作负责人和默认队伍人员 |
| 9 | 工作负责人 | 成员单选 | ✓ | 选队伍后自动带出，可手动修改为任意人员（全量不过滤），支持跨队伍指定 |
| 10 | 队伍人员 | 成员多选 | ✓ | 选队伍后自动带出默认成员，可增删任意人员（全量不过滤），支持跨队伍借人 |

> **自动带出逻辑**：选队伍后自动填充PC后台配置的默认负责人和默认成员列表。90%情况下不用修改，直接提交。需要跨队伍借人时，点击"添加"从全量人员中选择。

> **互斥规则**：工作负责人和队伍人员不可重复——同一人不能同时出现在两个角色中。工作负责人不属于人员池，在下游核查分工环节中，人员池仅由队伍人员构成。

**业务规则**：

- 表单为单页布局，全部字段一屏完成，不做分步
- V3.0创建与派发合并为一步操作（项目经理提交即完成派发，无审批环节）
- 传票号唯一性校验：提交时后端校验，重复则Toast提示"传票号已存在，请检查"，数据库唯一索引兜底
- 提交校验失败时：未填字段边框变红 + 自动滚动到第一个未填字段
- 提交前弹出确认弹窗，展示项目名称、检测日期、项目规模、工作负责人、队伍人员等关键信息摘要
- V3.0不做草稿保存（表单1分钟填完，投入产出比低），底座已实现，V3.1可启用
- V3.0不做派发后修改（填错时联系管理员后台修改），V3.1做项目经理自助撤回

**流程设计（Flowable单节点）**：

- V3.0流程定义（v1）：只有1个UserTask节点（`create_dispatch`），项目经理填完提交，流程直接结束，状态变为`dispatched`
- 提交后触发后处理链：①工单状态更新 ②`genStationRoomInfoNew()` 生成站房记录 ③队伍人员写入 `prj_project_member` 业务表
- V3.1通过Flowable流程版本管理部署v2，新增审批节点，已有v1工单不受影响

**接收确认（V3.0最小方案）**：

| 维度 | V3.0 | V3.1 |
|:---|:---|:---|
| 操作 | 查看工单摘要（只读）→ 确认接收 | 增加"拒绝"+ 退回修改 |
| 确认后 | 状态变为"已接收"，进入现场核查流程 | 同左 |
| 排期预览 | 不做 | 近期任务卡片 |

**工单列表**：

- 状态Tab筛选：全部 / 已派发 / 已接收
- 工单卡片展示：项目名称 + 状态标签 + 项目类型 + 规模（带标签）+ 检测日期 + 行政区划 + 工作负责人
- FAB创建按钮入口
- 空状态提示

**接收后数据流向**：

| 数据 | 下游消费 |
|:---|:---|
| 项目基础数据（名称、类型、传票号、区划、日期、低压新增） | 全流程使用，报告封面/表头 |
| 项目规模（站/房/合建/箱变） | `genStationRoomInfoNew()` 生成站房记录 → 建设情况确认 |
| 检测队伍 + 工作负责人 + 队伍人员 | 核查分工人员池（不含工作负责人）、报告签名区域 |

**V3.0与V3.1版本边界**：

| 能力 | V3.0 | V3.1 |
|:---|:---|:---|
| 创建 | 精简单页表单（13字段） | 同V3.0，可扩展 |
| 审批 | 不做 | 部署新版流程定义新增审批节点 |
| 派发 | 与创建合并一步完成 | 同V3.0 |
| 接收 | 确认接收 | 增加拒绝+退回 |
| 草稿 | 不做 | 启用底座已有能力 |

#### 1.1.0a 作业前准备

| 维度 | 说明 |
|:---|:---|
| **角色** | 工作负责人 |
| **时机** | 接收工单后，出发前或到达现场后（流程中第一个环节，实际时机由工作负责人自行决定） |
| **做什么** | 对材料、人员、项目层面的准备情况做合规确认（过门页） |
| **版本归属** | V3.0 |

**业务定位**：合规过门页。日常使用中90%以上是一键全合格通过，核心价值在于**合规留痕**而非操作本身。独立存在的意义：①新人引导（出发前的准备工作检查清单）②售前展示（向供电公司领导展示全链路规范性覆盖）。

**检查项设计（3组7项）**：

*材料准备（4项）*：

| 序号 | 检查项 | 对应规范 | 操作方式 |
|:---|:---|:---|:---|
| 1 | 纸质记录单 | 纸质材料 | 合格/不合格 toggle |
| 2 | 检测仪器 | 试验仪器 | 合格/不合格 toggle |
| 3 | 安全工器具 | 安全工器具 | 合格/不合格 toggle |
| 4 | 工程车辆 | 车辆准备 | 合格/不合格 toggle |

*人员准备（2项）*：

| 序号 | 检查项 | 对应规范 | 操作方式 |
|:---|:---|:---|:---|
| 5 | 人员情况 | 人员安排（资质+人数） | 合格/不合格 toggle |
| 6 | 主业联系 | 勘察准备 | 合格/不合格 toggle |

*项目准备（1项）*：

| 序号 | 检查项 | 对应规范 | 操作方式 |
|:---|:---|:---|:---|
| 7 | 配合事项要求 | 配合事项 | 合格/不合格 toggle + 一键复制配合事项文本 |

> **配合事项复制功能**：点击展开结构化文本卡片（含现场条件、资料准备、电源与工具、人员等要求），底部提供"复制"按钮。工作负责人复制后切换到微信粘贴发送给供电公司专职。复制与合格判定是两个独立动作，互不影响。V3.0使用固定模板，V3.1可按项目类型配置不同模板。

**业务规则**：

- **一键全部合格**：底部快捷操作入口，点击后7项全部设为合格。若随后手动将某项改为不合格，按钮恢复可点击
- **不能默认合格**：必须由工作负责人主动点击确认（华工确认："和供电公司领导交代不了"）
- 7项全部判定后才激活提交按钮，防止遗漏
- **全合格时不弹确认弹窗**，直接提交（减少一步操作）
- 存在不合格项时弹窗二次确认："存在N项不合格，确认提交？"
- **不合格不中止流程**：不合格仅记录留存，工单正常流转到下一环节
- 分组进度计数：每组右上角显示完成进度（如"0/4"），全组完成后计数变绿
- 页面顶部展示项目摘要（项目名称+检测日期），从上游工单数据读取
- **中断不保留**：退出页面后重新进入需重新操作（2步5秒的过门页，中间状态保存无必要）
- **提交后不可修改**：一次性过门，V3.0不做回退
- 省检/市检V3.0不差异化
- 检查项提示文案（规范原文、操作指引）V3.0不做，规划V3.1

**提交数据**：7项检查结果（各项合格/不合格状态）+ 提交时间 + 操作人（工作负责人）

**流程节点（Flowable）**：

| 节点 | 节点Key | 类型 | 操作人 | 权限 | 行为 |
|:---|:---|:---|:---|:---|:---|
| 作业前准备 | `pre_work` | UserTask | 工作负责人 | 工作负责人可操作；其他人员只读 | 提交后流转到"现场准备"环节 |

**数据流向**：

| 方向 | 数据 | 说明 |
|:---|:---|:---|
| 上游输入 | 项目名称、检测日期 | 页面顶部项目摘要展示 |
| 上游输入 | 工作负责人指派 | 确定本环节操作角色 |
| 下游输出 | 准备确认状态 + 确认时间戳 | 流程引擎放行，允许进入现场准备环节 |

#### 1.1.0b 现场准备

| 维度 | 说明 |
|:---|:---|
| **角色** | 工作负责人 |
| **时机** | 到达现场后 |

**与作业前准备的闭环关系**：

#### 1.1.1 建设情况确认

| 维度 | 说明 |
|:---|:---|
| **角色** | 工作负责人 |
| **时机** | 现场准备完成后，到达现场对照报检资料确认站房信息 |
| **做什么** | 核实站房数量/编号/类型，确认配套设施，确认进线电缆段数——为下游全部环节建立站房数据基础 |
| **版本归属** | V3.0 |

**业务定位**：建设情况确认的目的不是填站房档案，而是为现场核查建立数据基础。页面做三件事：①站房数量/类型/编号确认 ②室外配套设施勾选 ③进线电缆段数确认。

**与"建设情况核查"的区别**：

| | 建设情况确认（本节） | 建设情况核查（1.1.3核查执行中的一项） |
|:---|:---|:---|
| 时间 | 核查分工**之前** | 核查分工**之后** |
| 角色 | 工作负责人 | 试验人员 |
| 内容 | 填写站房基础信息 | 逐站房拍照（外景+内景）+ 判定 |
| 数据关系 | 产生站房列表数据 | 消费站房列表，追加拍照和判定 |

**设计方案：轻量建站（从"全量录入"转为"核实+修正"）**：

2.4系统要求16字段×N站房全量录入（38+次点击），但源码追踪发现：16个字段中**现场核查只用3个**（站房名称、站房类型、进线电缆数量）。3.0系统只保留核查必须的字段，其余延后到抽检对象确认时补填。

**信息时间线**：

| 时间节点 | 已知信息 | 3.0处理方式 |
|:---|:---|:---|
| 工单创建时（出发前） | 站房数量和类型 | 系统自动生成站房记录，预设默认编号 |
| 到达现场当天 | 抽检规则（供电公司当天发送Word表格截图） | 后续环节处理，本页面不涉及 |
| 工作负责人到场后 | 站房实际编号（现场门牌）、进线电缆数量 | 核实/修正系统预设值 |
| 抽检对象确认时 | 高压柜类型、PT类型、设备编号等 | 此时再填详细站房属性+设备信息 |

**每站房涉及的字段（仅2个）**：

| 字段 | 操作方式 | 默认值 | 说明 |
|:---|:---|:---|:---|
| 站房编号 | 点击数字弹出1-10面板选择 | `{序号}`（数字） | 大多数是数字（1#、2#），面板底部兜底"输入其他编号"入口 |
| 站房类型 | 点击类型标签弹出选择 | 工单创建时确定 | 通常不需改，类型错误时才调整 |

> 同类型下编号唯一性校验，重复则Toast提示。

**配套设施确认（有/无 双按钮必选）**：

| 字段 | 默认值 | 下游影响 |
|:---|:---|:---|
| 充电桩设施 | 无默认值（必须选） | 选"有"→ 核查分工中出现"充电设施"核查任务（`chargingSetCompleteFlag`） |
| 居民配套设施 | 无默认值（必须选） | 选"有"→ 核查分工中出现"居民设施"核查任务（`residentialFacilityCompleteFlag`） |

> 为什么不用勾选框：勾选框的"未勾选"无法区分"用户选了无"还是"用户没看到"。有/无双按钮保证用户一定看过并做了选择。

**进线电缆段数确认（Radio必选）**：

| 字段 | 操作方式 | 默认值 | 下游影响 |
|:---|:---|:---|:---|
| 进线电缆段数 | Radio（1段/2段/3段） | 2段 | 1段=每站房4项电源核查 / 2段=6项 / 3段=8项 |

> 全局设置（项目级），不按站房独立设置。每个选项右侧显示影响说明。

**站房状态设计**：

| 状态 | 视觉表现 | 触发条件 |
|:---|:---|:---|
| 待确认 | 左侧橙色竖条 | 初始状态和新增状态 |
| 已确认 | 左侧绿色竖条 | 点击行空白区域确认，或修改编号后自动确认 |

**交互规范（Material Design 3）**：

| 交互 | 方式 |
|:---|:---|
| 确认编号 | 点击行空白区域（橙→绿） |
| 改编号 | 点击数字 → Bottom Sheet数字面板（1-10网格 + 兜底输入） |
| 删除站房 | 长按行 → 操作菜单 → 二次确认弹窗（至少保留1个站房） |
| 新增站房 | 底部"+ 新增站房"按钮 → Bottom Sheet类型选择 → 自动弹出数字面板引导选编号 |

**业务规则**：

- 站房列表按类型分组展示（开关站/配电室/合建/箱变），空分组不显示
- 新增站房后自动弹出数字面板引导选编号
- 提交前三段校验：①站房编号全部已确认 ②配套设施全部已选 ③进线段数已选。校验不通过自动滚动到未完成区域
- 底部进度统计："已确认 X/Y 个站房"
- **中断不保留**：退出重新进入需重新操作
- **提交后不可修改**：站房编号作为全流程数据源，修改影响面过大
- **按站房粒度逐个解锁下游**（已确认设计方向）：1号站房核实完成 → 立即解锁1号站房的核查分工和核查执行，不等其他站房

**提交后触发的后处理链**：

```
建设情况确认"提交"
  │
  ├── verifyComplete() — 校验
  │     ├── 所有站房 isComfired == "1"
  │     └── 规模匹配（弱同步，不一致时自动更新项目规模）
  │
  └── confirmStationRoomInfo() — 后处理
        ├── syncCheckInfo() — 从配置表生成核查项快照
        │     ├── 8大核查项（Set层）
        │     │     ├── "充电设施"核查 → chargingSetCompleteFlag控制

        │     ├── 核查分组（Group层）→ 按站房类型选对应模板
        │     └── 核查数据项（Item层）→ 按进线数量裁剪电源核查子项
        ├── 更新项目状态 → IN_CHECKING

```

#### 1.1.2 核查分工

| 维度 | 说明 |
|:---|:---|
| **角色** | 工作负责人 |
| **时机** | 建设情况确认之后 |
| **做什么** | 将6~8项核查任务按人分配给4名试验人员（无"组"概念） |
| **版本归属** | V3.0 |

**设计决策**：

| 决策 | 结论 | 依据 |
|:---|:---|:---|
| 分工模式 | **按人分配**，无"一组/二组"概念 | 张工（省检）"核查不存在小组"；华工（市检）"不需要默认分工" |
| 候选人 | 4名试验人员（工作负责人不参与核查） | 工作负责人同时在做抽检对象确认（抽柜子） |
| 一人多项 | ✓ 支持 | 省检场景中一人可负责多项核查 |
| 一项多人 | ✓ 支持，但必须有1个负责人 | 负责人唯一，其他人可协同拍照 |
| 全员可见 | 所有人看到全部任务 | 干完的人可直接进入别人的任务帮忙 |
| 必须全部分配 | ✓ 提交前强制校验 | "现场的人很容易点漏，系统要强制" |
| 默认推荐 | V3.0不做，V3.1做后台配置默认分工模板 | V3.0先让用户熟悉交互 |

**核查任务列表**：

| 物理位置 | 核查项 | 是否固定存在 |
|:---|:---|:---|
| 站房内 | 电源核查 | ✓ 固定 |
| 站房内 | 建设情况 | ✓ 固定 |
| 站房内 | 送电痕迹 | ✓ 固定 |
| 站房内/外 | 临电电源 | ✓ 固定 |
| 站房外 | 居民入住 | ✓ 固定 |
| 站房外 | 安全风险 | ✓ 固定 |
| 站房外 | 充电设施 | ⚡ 条件出现（建设情况确认勾选"有充电桩设施"） |
| 站房外 | 居民设施 | ⚡ 条件出现（建设情况确认勾选"有居民配套设施"） |

> 任务列表 = 基础6项（固定）+ 0~2项（条件出现）= 6~8项。物理位置分类仅供工作负责人分工参考，不作为系统硬约束。

**交互设计——"先想人，再分活"**：

页面结构：4张人员卡片纵向排列，每张卡片下方显示该人负责的任务标签（chip）。

- 点击某人的"+ 添加" → Bottom Sheet弹出未分配任务列表（可多选）→ 确认后任务标签出现在该人卡片下

#### 1.1.3 核查执行（全部核查项并行）

| 维度 | 说明 |
|:---|:---|
| **角色** | 4名试验人员（被分配核查任务的人） |
| **前置条件** | 核查分工完成 |
| **做什么** | 全部6~8项核查并行执行（无依赖、无前置关卡），拍照留证+整项判定 |
| **版本归属** | V3.0 |

> **重大设计变更**：2.4系统的"电源核查强制关卡"（必须完成后才能做其他核查）已取消。依据：张工（省检）确认"省检现场本来就是并行作业，强制关卡反而耽误事"。电源核查的安全优先级通过现场管理（工作负责人口头安排）保证，不通过系统强制。

**判定模型——拍照只留证，判定只有一层**：

- **拍照是留证**：回答"现场长什么样"，不回答"合不合格"。任何人都能拍
- **判定是结论**：负责人看完全部照片后给整个核查项一次最终结论（合格/不合格）。仅负责人有权（后端校验）
- 去掉2.4系统的逐照片判定和逐站房判定（两层判定增加操作负担且语义矛盾）

**两种核查任务类型**：

| 类型 | 核查项 | 3.0交互 |
|:---|:---|:---|
| **📷 拍照型**（5项） | 电源核查、建设情况、居民入住、充电设施、居民设施 | 进入拍照模式 → 拍照（每slot最多5张）→ 全部拍完后底部展开判定表单 |
| **📋 直接判定型**（3项） | 送电痕迹、临电电源、安全风险 | 弹出Bottom Sheet → 直接判定。不合格时补拍留证照片 |

**拍照型任务的交互流程**：

```
进入拍照模式（同一手机壳内，不调用原生相机）
    │
    ├── 上半屏：自定义取景窗（标签显示当前拍照项名称）
    ├── 中部：拍照项列表（选中已拍项展开照片网格，可逐张删除）
    └── 底部：快门 + 进度点阵
    │
    ▼
按快门拍照（每项最多5张）
    │  拍满5张 → 自动跳下一个空项
    │  每张照片带操作人ID+时间戳
    │
    ▼
全部拍照项至少1张 → 快门区域消失，底部原地展开判定表单（不跳页）
    │
    │  负责人视角：radio（合格/不合格）+ [取消] [提交判定]
    │  非负责人视角："等待XX进行判定" + [返回任务列表]
```

**不合格流程（拍照型）**：

```
选"不合格" → Bottom Sheet弹出（不压缩页面）
    │  勾选不合格原因（至少1项，5项固定列表）
    │  [取消] [去拍摄留证照片 →]
    ▼
回到取景窗拍留证照片（红色标签区分，复用同一套快门，最多5张）
    ▼
回到判定表单 → 提交判定
```

**直接判定型任务**：点击任务 → Bottom Sheet判定面板（合格/不合格 radio）→ 选不合格后展开原因勾选+留证照片入口 → 提交

**Slot列表动态生成**：
- 电源核查slot数 = 进线段数决定（1段=4项 / 2段=6项 / 3段=8项）
- 建设情况核查slot数 = 站房数量 × 2（外景+内景）
- 其他核查项slot数 = 固定或基于配置

**多人协同模型**：

| 操作类型 | 谁能做 | 冲突处理 |
|:---|:---|:---|
| 拍照 | 所有人 | 追加操作，每张照片带操作人ID+时间戳，多人拍的全部保留 |
| 判定 | 仅负责人 | 后端校验操作人=负责人，仅1人有权 |

同步策略：
- **有网**：拍照即上传，每10-15秒轮询增量同步（`GET /items?since=lastSyncTime`）
- **无网**：全部操作存本地，恢复网络后一次性批量上传合并
- 不做弱网重试/断点续传，二元化处理

**业务规则**：
- **中间状态必须保留**：照片拍一张存一张（存本地），不能因APP退出/崩溃而丢失（与前面"过门页"不同——现场实拍不可重来）
- **取景窗为自定义组件**：不调用原生相机（保持交互上下文，底部始终显示拍照项列表和进度）
- 每slot照片上限5张（V3.0统一上限）
- 合格路径极简（radio选合格→提交），不合格复杂内容用Bottom Sheet承载
- 不合格留证拍照复用页面已有取景窗和快门
- 8项全部完成的校验：拍照型5项所有slot≥1张+负责人已判定，判定型3项负责人已判定，全部完成才放行
- 不合格原因V3.0使用固定列表（5项），V3.1可扩展为PC端配置

**电源核查子项结构（标准项目，按位置分组）**：

| 分组 | 子项 | 操作 | 动态条件 |
|:---|:---|:---|:---|

> 低压新增项目走简化模式（仅确认新增电缆未搭接到带电低压柜上）。

# 九、非功能需求

## 1. 网络策略（有网/无网）

> 不做"弱网"中间态区分。系统只识别两种网络状态：有网、无网。

| 状态 | 定义 | 系统行为 |
|:---|:---|:---|
| **有网** | APP与服务器可正常通信 | 照片本地优先存储+后台异步上传；多人协作轮询同步；提交时校验全部数据已上传 |
| **无网** | APP与服务器无法通信 | 照片存本地；操作存本地队列；不做多人协作同步；恢复网络后自动上传+合并 |

**技术策略**：本地优先写入——拍照、判定等操作先写本地，后台异步同步服务器。用户操作零等待。

**照片上传机制**：
- 拍照后立即存入本地（毫秒级响应），页面即时显示缩略图
- 后台队列异步上传，有网时自动进行
- 无网时暂存，恢复网络后自动续传
- 最终提交时校验：所有照片上传完成才允许提交

**多人协作同步**：
- 仅在有网时通过轮询（通用同步服务）实时同步操作状态
- 无网时各自独立操作，恢复网络后自动合并
- 同步的是状态信息（几十字节），不是照片本身
